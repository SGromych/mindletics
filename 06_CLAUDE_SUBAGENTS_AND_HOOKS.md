# Mindletics — субагенты и хуки для Claude Code

## Зачем это нужно

Для проекта Mindletics не нужен слишком сложный multi-agent orchestration. Это не гигантская enterprise-система, а понятный MVP: планшетный веб-сайт с несколькими экранами, фиксированным сценарием event, когнитивными тестами, live-табло и лидербордами.

Поэтому для Claude Code лучше использовать **небольшой, практичный набор субагентов** и **минимальный набор хуков**, которые помогают:
- не ломать архитектуру;
- не забывать про требования планшетного UI;
- не портить модель данных;
- не разъезжаться в логике этапов;
- не вносить хаос в когнитивные тесты;
- не забывать обновлять README и инструкции.

Ниже — рекомендуемая схема.

---

# 1. Какие субагенты нужны

## Общий принцип

Для этого проекта достаточно **5 специализированных субагентов**:

1. **architect**
2. **frontend-ui**
3. **backend-data**
4. **test-engine**
5. **qa-review**

Этого достаточно. Больше делать не нужно, иначе будет лишняя сложность.

---

## 1.1. Subagent: architect

### Назначение
Отвечает за общую структуру проекта и архитектурные решения.

### Что делает
- следит, чтобы проект оставался простым;
- удерживает выбранный стек;
- следит за структурой папок;
- не допускает ненужного усложнения;
- проверяет, что frontend, API, Prisma и данные тестов согласованы между собой;
- следит за тем, чтобы сценарий event оставался фиксированным, как в ТЗ.

### Когда вызывать
- при инициализации проекта;
- при создании структуры папок;
- при добавлении новых сущностей;
- при изменении flow прохождения;
- перед крупным рефакторингом.

### Что ему поручать
- придумать файловую структуру;
- проверить, не стал ли проект слишком сложным;
- проверить соответствие ТЗ;
- согласовать контракты между frontend и backend.

### Пример задач для него
- “Проверь, не переусложнена ли архитектура MVP.”
- “Согласуй структуру проекта и API.”
- “Проверь, соответствует ли flow attempt фиксированному сценарию event.”

---

## 1.2. Subagent: frontend-ui

### Назначение
Отвечает за интерфейсы, взаимодействие пользователя и планшетный UX.

### Что делает
- проектирует и реализует экраны;
- следит за крупными кнопками и удобством tap-взаимодействия;
- делает компоненты для тестов;
- обеспечивает понятный UI для участника;
- следит, чтобы live-табло и лидерборды были читаемыми;
- следит, чтобы не появлялись мелкие кликабельные элементы.

### Когда вызывать
- при создании новых экранов;
- при доработке планшетного интерфейса;
- при проектировании когнитивных тестов;
- при правке layout и адаптивности.

### Что ему поручать
- сделать главный экран;
- сделать форму регистрации участника;
- сделать экран прохождения;
- сделать интерфейсы для когнитивных блоков;
- сделать live-табло;
- упростить UI, если он стал перегруженным.

### Пример задач для него
- “Сделай экран attempt удобным для 10-дюймового планшета.”
- “Упрости визуально live-табло.”
- “Переделай компоненты ответа так, чтобы ими было удобно пользоваться пальцем.”

---

## 1.3. Subagent: backend-data

### Назначение
Отвечает за данные, API, сохранение результатов и логику переходов.

### Что делает
- проектирует Prisma schema;
- реализует API routes;
- управляет состоянием attempt;
- сохраняет stage results и raw answers;
- реализует leaderboards;
- отвечает за polling/live endpoints;
- следит за согласованностью статусов.

### Когда вызывать
- при работе с базой данных;
- при создании API;
- при реализации scoring;
- при реализации переходов между этапами;
- при сохранении логов ответов.

### Что ему поручать
- написать Prisma schema;
- реализовать start / next-level / abort;
- реализовать leaderboard query;
- сделать endpoint для live-табло;
- проверить корректность сохранения raw answers.

### Пример задач для него
- “Сделай Prisma schema для Event, Participant, Attempt, StageResult.”
- “Реализуй API перехода на следующий этап.”
- “Проверь, что Abort и Finished корректно отражаются в БД.”

---

## 1.4. Subagent: test-engine

### Назначение
Отвечает именно за когнитивные тесты как мини-движок внутри проекта.

### Что делает
- стандартизирует формат JSON-заданий;
- следит, чтобы все тесты были только tap-based;
- реализует проверку правильности ответов;
- считает correct/wrong/skipped;
- определяет, как рендерить разные типы заданий;
- обеспечивает единый контракт между тестовыми данными и UI.

### Когда вызывать
- при проектировании структуры тестовых JSON;
- при добавлении новых типов заданий;
- при реализации блоков logic/memory/reaction/visual_final;
- при проверке scoring по заданиям.

### Что ему поручать
- придумать единый формат test item;
- реализовать движок проверки ответов;
- привести все типы заданий к единой модели;
- проверить, что все задания работают без текстового ввода.

### Пример задач для него
- “Предложи единый формат JSON для всех тестов.”
- “Проверь, что все задания можно пройти только касаниями.”
- “Сделай нормализованную модель логирования ответов.”

---

## 1.5. Subagent: qa-review

### Назначение
Финальная проверка изменений.

### Что делает
- читает дифф;
- ищет регрессии;
- проверяет соответствие ТЗ;
- смотрит, не сломались ли экраны;
- ищет дублирование кода;
- проверяет, что README и инструкции не устарели;
- следит, что проект можно запустить.

### Когда вызывать
- после значимых изменений;
- перед завершением задачи;
- перед коммитом;
- после рефакторинга.

### Что ему поручать
- code review;
- functional review;
- UX review;
- review на соответствие требованиям MVP.

### Пример задач для него
- “Проверь весь diff и найди регрессии.”
- “Проверь проект на соответствие ТЗ Mindletics.”
- “Проверь, можно ли реально пройти весь event от начала до конца.”

---

# 2. Какие субагенты НЕ нужны

Для этого проекта пока не нужны:
- отдельный DevOps-agent;
- отдельный Security-agent;
- отдельный Analytics-agent;
- отдельный Performance-agent;
- отдельный Prompt-agent;
- отдельный Documentation-agent.

Почему:
- проект пока маленький;
- деплой можно сделать простым;
- безопасность пока базовая;
- аналитика и observability не являются ядром MVP;
- документацию может поддерживать qa-review или architect.

# 3. Рекомендуемое распределение задач между субагентами

## Этап 1. Старт проекта
Использовать:
- architect
- backend-data

### Что делать
- создать структуру проекта;
- выбрать стек;
- создать Prisma schema;
- подготовить базовый routing.

---

## Этап 2. Базовый UI
Использовать:
- frontend-ui

### Что делать
- главный экран;
- экран event;
- экран participant registration;
- экран attempt;
- live-табло;
- лидерборды.

---

## Этап 3. Когнитивные тесты
Использовать:
- test-engine
- frontend-ui
- backend-data

### Что делать
- структура JSON-заданий;
- компоненты тестов;
- логирование ответов;

---

## Этап 4. Сквозной flow

- переходы между этапами;

## Этап 5. Финальная стабилизация
Использовать:
- qa-review

### Что делать
- проверить сценарии;
- проверить UX;
- проверить API;
- проверить README;
- проверить seed и запуск.

# 4. Какие хуки нужны

## Общий принцип

Хуков тоже не нужно делать слишком много. Достаточно **6 практических хуков**, которые дисциплинируют работу Claude Code.

---

## 4.1. Hook: before-large-change

### Когда срабатывает
Перед крупным изменением, если Claude собирается:
- создать много файлов;
- рефакторить несколько модулей;
- менять Prisma schema;
- переписывать flow прохождения.

### Что должен делать
Claude перед началом обязан кратко написать:
1. что именно он хочет изменить;
2. какие файлы затронет;
3. какие риски есть;
4. как проверит, что всё не сломалось.

### Зачем
Чтобы не было “silent chaos”, когда агент начинает масштабный рефакторинг без плана.

### Что проверять
- есть ли план;
- не нарушается ли простота MVP;
- не меняется ли scope без спроса.

---

## 4.2. Hook: before-schema-change

### Когда срабатывает
Перед изменением `schema.prisma`, SQL-миграций или моделей данных.

### Что должен делать
Claude обязан:
1. описать, зачем меняется модель;
2. перечислить затрагиваемые сущности;
3. указать, какие API и страницы это затронет;
4. проверить обратную совместимость seed и существующего кода.

### Зачем
Чтобы не сломать flow проекта из-за неаккуратной правки БД.

---

## 4.3. Hook: before-ui-change

### Когда срабатывает
Перед изменением экранов участника, тестов или live-табло.

### Что должен делать
Claude должен проверить:
1. удобно ли это для планшета;
2. не стало ли слишком мелко;
3. можно ли пройти сценарий пальцем;
4. не исчезли ли важные действия Start / Next level / Abort.

### Зачем
Потому что главный риск проекта — сделать UI, который красиво выглядит на десктопе, но неудобен на планшете.

---

## 4.4. Hook: before-test-content-change

### Когда срабатывает
При изменении структуры тестовых заданий, формата JSON или движка проверки ответов.

### Что должен делать
Claude обязан:
1. проверить, что задания остаются tap-only;
2. проверить, что нет текстового ввода;
3. проверить, что везде 3–5 вариантов или работа с визуальными объектами;
4. проверить, что сохраняются raw answers;
5. проверить, что scoring по тесту не сломан.

### Зачем
Чтобы тесты не расползлись в разные форматы и не потеряли единообразие.

---

## 4.5. Hook: after-change-self-review

### Когда срабатывает
После каждого заметного изменения.

### Что должен делать
Claude обязан коротко сделать self-review:
1. что изменено;
2. какие сценарии теперь работают;
3. что ещё не проверено;
4. какие риски остались.

### Зачем
Чтобы сохранялась прозрачность прогресса.

---

## 4.6. Hook: before-finish-task

### Когда срабатывает
Перед тем как Claude говорит, что задача выполнена.

### Что должен делать
Claude должен проверить checklist:
- проект запускается;
- схема данных актуальна;
- README обновлён;
- новые env переменные описаны;
- seed не сломан;
- UI соответствует планшетному сценарию;
- live-табло работает;
- лидерборды работают;
- raw answers сохраняются;
- Next level и Abort защищены от ошибок.

### Зачем
Чтобы не было формального “готово”, когда half-done результат ещё не собран в рабочее состояние.

---

# 5. Практическая конфигурация поведения Claude Code

Ниже — практический набор правил, который можно дать Claude Code как рабочую политику проекта.

## Основные правила

### Правило 1
Если изменение затрагивает архитектуру, структуру проекта, flow attempt или модель данных — сначала подключай **architect**.

### Правило 2
Если изменение затрагивает экраны, планшетный UX, компоненты ответов, layout — подключай **frontend-ui**.

### Правило 3
Если изменение затрагивает Prisma, API, сохранение результатов, статусы, leaderboard — подключай **backend-data**.

### Правило 4
Если изменение затрагивает типы когнитивных заданий, JSON-тесты, проверку ответов, raw logs — подключай **test-engine**.

### Правило 5
Перед завершением значимой задачи подключай **qa-review**.

# 6. Рекомендуемые шаблоны вызова субагентов

## Шаблон для architect
“Проверь архитектуру текущего изменения для проекта Mindletics. Следи, чтобы MVP оставался простым: один Next.js проект, PostgreSQL, Prisma, без сложной авторизации, без лишней гибкости event.”

## Шаблон для frontend-ui
“Проверь и улучши этот экран с точки зрения планшетного UX. Участник работает пальцем, кнопки должны быть крупные, интерфейс должен быть понятен после физической нагрузки.”

## Шаблон для backend-data
“Проверь модель данных и API для Mindletics. Убедись, что корректно сохраняются attempt, stage results и raw answers, а лидерборды строятся по правилам MVP.”

## Шаблон для test-engine
“Проверь когнитивный блок Mindletics. Все задания должны быть tap-only, без текста и без печати. Убедись, что формат заданий единый и результаты можно логировать одинаково.”

## Шаблон для qa-review
“Сделай финальный review diff для Mindletics: найди регрессии, проверь соответствие ТЗ, проверь UX для планшета и проверь, что приложение можно реально запустить.”

# 7. Минимальный набор хуков, который реально стоит включить

Если нужен совсем короткий practical set, то оставьте только это:

1. **before-large-change**
2. **before-schema-change**
3. **before-ui-change**
4. **before-test-content-change**
5. **after-change-self-review**
6. **before-finish-task**

Это оптимальный минимум.

---

# 8. Итоговая рекомендация

Для проекта Mindletics в Claude Code лучше использовать **малый, дисциплинированный набор**:

## Субагенты
- architect
- frontend-ui
- backend-data
- test-engine
- qa-review

## Хуки
- before-large-change
- before-schema-change
- before-ui-change
- before-test-content-change
- after-change-self-review
- before-finish-task

Такой набор:
- не переусложняет работу;
- помогает держать проект в рамках MVP;
- снижает риск хаоса в коде;
- подходит именно для планшетного веб-проекта с фиксированным сценарием соревнования.
