# Mindletics — задание для Claude Code

## Контекст проекта

Нужно сделать простой веб-сайт для планшета для проекта **Mindletics**.

Mindletics — это соревнования, где у участника чередуются:
1. физическое упражнение;
2. интеллектуальный тест на планшете.

Сайт должен быть **максимально простым**, без сложной авторизации, без ролей, без избыточной гибкости. Это не универсальная платформа-конструктор, а рабочий MVP под **один сценарий соревнования**.

## Важные ограничения

### Что нужно сделать просто
- Это именно **сайт**, открывающийся в браузере планшета.
- Сложная авторизация **не нужна**.
- Роли **не нужны**.
- Один и тот же интерфейс может использоваться:
  - для регистрации event,
  - для регистрации участника,
  - для прохождения,
  - для показа live-табло,
  - для просмотра лидербордов.
- Пока поддерживается **только один сценарий event**.
- Гибкий конструктор event **не нужен**.
- Рандомизация заданий **не нужна**.
- Уровни сложности **не нужны**.
- Лимиты времени на этапы **не нужны**.
- Обработка зависания/пропуска этапа **не нужна**.
- Все участники получают **одинаковые задания**.
- Нужно сохранять **сырые ответы** участника по тестам для логирования.

### Что нельзя делать
- Не добавлять сложную RBAC-модель.
- Не делать многопользовательскую админку с правами.
- Не делать отдельный мобильный app.
- Не делать слишком сложный конструктор тестов.
- Не делать голосовой ввод, печать слов, свободный текст.
- Не усложнять серверную архитектуру без необходимости.

## Бизнес-логика

### Как проходит соревнование

1. Организатор создаёт event:
   - название зала,
   - название event,
   - дата.

2. На планшете регистрируется участник:
   - ФИО одной строкой **или** ник,
   - пол,
   - возраст,
   - выбор event из существующих.

3. После регистрации участнику показывается:
   - название event,
   - зал,
   - дата,
   - план прохождения,
   - кнопка **Start**.

4. После нажатия **Start**:
   - запускается **общий таймер**;
   - участник уходит выполнять физическое упражнение;
   - когда возвращается, сам нажимает **Next level**;
   - открывается интеллектуальный тест;
   - по завершении теста участник нажимает **Next level**;
   - запускается следующий физический этап;
   - и так до конца сценария.

5. Если участник нажимает **Abort**:
   - попытка завершается;
   - в итоговом лидерборде отмечается, что участник не завершил;
   - данные о статусе сохраняются.

6. После завершения всех этапов:
   - результат сохраняется в БД;
   - участник попадает в лидерборд.

## Сценарий event (фиксированный)

### Этап 1
Физическое упражнение: **1 км гребля**

### Этап 2
Интеллект-тест: **логика / абстрактное мышление**

### Этап 3
Физическое упражнение: **40 становых тяг (60 кг мужчины / 40 кг женщины) + 40 берпи боком через штангу**

### Этап 4
Интеллект-тест: **память**

### Этап 5
Физическое упражнение: **3 раунда: 15/12 кал байк + 10 прыжков через коробку**

### Этап 6
Интеллект-тест: **скорость реакции / когнитивный контроль**

### Этап 7
Физическое упражнение: **300 м челночный бег**

### Этап 8
Интеллект-тест: **финальный визуально-логический блок**

## Главная идея по тестам

Для MVP использовать **только планшетные тесты**, где участник:
- нажимает пальцем;
- выбирает ответ из **3–5 вариантов**;
- взаимодействует только через **tap**;
- ничего не печатает;
- не вводит текст;
- не говорит в микрофон.

### Важно
Во всех когнитивных блоках должны быть задания, которые:
- короткие;
- понятные;
- легко автоматизируются;
- объективно оцениваются;
- сохраняют сырые ответы;
- подходят для экрана планшета.

## Набор тестов для MVP

Нужно реализовать **4 когнитивных блока**, каждый примерно по **5–6 заданий**.

### Когнитивный блок 1. Логика / абстрактное мышление
Типы заданий:
1. **Числовые ряды**
   - участнику показывается последовательность;
   - нужно выбрать следующий элемент из 4 вариантов.

2. **Фигурные последовательности**
   - ряд фигур меняется по правилу;
   - нужно выбрать следующую фигуру из 4 вариантов.

3. **Простые матрицы**
   - 2x2 или 3x3, где не хватает элемента;
   - нужно выбрать недостающий элемент из 4 вариантов.

На блок: 5–6 заданий.

### Когнитивный блок 2. Память
Типы заданий:
1. **Запоминание объектов**
   - показывается набор из 5–6 объектов;
   - затем участник выбирает, какие были, из большего набора.

2. **Запоминание позиций в сетке**
   - на 3–4 секунды подсвечиваются клетки;
   - потом нужно нажать те же клетки.

3. **Последовательность объектов**
   - несколько фигур показываются по очереди;
   - потом нужно нажать их в том же порядке.

На блок: 5–6 заданий / раундов.

### Когнитивный блок 3. Скорость реакции / когнитивный контроль
Типы заданий:
1. **Choice reaction**
   - загорелся цвет/символ;
   - нужно нажать соответствующую кнопку.

2. **Go / No-Go**
   - если целевой стимул — нажать;
   - если нецелевой — не нажимать.

3. **Stroop с кнопками**
   - показывается слово цвета, окрашенное другим цветом;
   - нужно нажать кнопку, соответствующую **цвету текста**, а не смыслу слова.

На блок: 5–6 заданий / стимулов.

### Когнитивный блок 4. Финальный визуально-логический блок
Чтобы не делать текстовую беглость и не вводить слова, вместо неё делаем ещё один полностью планшетный блок:
1. **Перцептивная скорость**
   - найти целевой символ среди похожих;
   - выбрать правильный вариант.

2. **Сравнение пар**
   - определить, одинаковы ли два набора символов/фигур;
   - нажать «Да» или «Нет».

3. **Короткие визуальные аналогии**
   - визуальная пара и ещё одна пара с пропуском;
   - выбрать недостающий элемент из 4 вариантов.

На блок: 5–6 заданий.

## Подсчёт результата

Нужна простая модель.

### Предлагаемая схема scoring
- за каждый правильный ответ начисляются баллы;
- при равенстве правильных ответов выше тот, у кого меньше общее время.

Простой вариант:
- `total_correct` = сумма правильных ответов;
- `total_time_sec` = время от Start до Finish;
- сортировка:
  1. по `total_correct` по убыванию;
  2. при равенстве — по `total_time_sec` по возрастанию.

Также можно считать:
- `total_wrong`
- `accuracy_percent`

Но основная сортировка должна быть максимально простой и прозрачной.

## Защита от ошибок пользователя

Так как **Next level нажимает сам участник**, нужно сделать защиту:
- после нажатия `Next level` показать короткое подтверждение;
- или блокировать повторное нажатие на 1–2 секунды;
- нельзя допустить случайного двойного перехода;
- для `Abort` обязательно сделать подтверждение:
  - «Вы точно хотите завершить попытку?»

## Рекомендуемая схема реализации

Нужно предложить и реализовать **простую** архитектуру.

### Вариант, который предпочтителен
Сделать один репозиторий с двумя частями:
- frontend;
- backend.

### Предпочтительный стек
#### Frontend
- Next.js
- TypeScript
- Tailwind CSS

#### Backend
- Next.js API routes **или** простой Express/Nest backend

#### Database
- PostgreSQL

#### Real-time
- самый простой способ:
  - polling раз в 2–3 секунды
  - **или** WebSocket, если получится без перегруза

### Упрощение
Если WebSocket усложняет MVP, можно сделать **обычный polling**:
- live-страница перезапрашивает данные каждые 2 секунды.

Это допустимо для первой версии.
